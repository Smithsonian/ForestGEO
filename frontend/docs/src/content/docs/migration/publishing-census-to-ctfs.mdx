---
title: Publishing a Census to CTFSWeb
description: How to use the Publish census button to load a finished ForestGEO census into the on-prem CTFSWeb MySQL database.
---

The **Publish census** button generates a self-contained `.sql` artifact that loads a finished ForestGEO census into the on-prem CTFSWeb MySQL database. Operators download the artifact, run it against their CTFS destination, and the validated measurements land in `Tree`, `Stem`, `DBH`, and `DBHAttributes` — followed by an automatic `ViewFullTable` rebuild.

This page covers when to use the feature, what each option does, what the artifact contains, how to run it, and the common ways it can refuse to publish.

---

## When to use Publish census vs. the legacy CSV export

Both pathways are still supported. They serve different consumers.

| Use the **Publish census** button (this page) when… | Use the legacy **CTFSWebForms** CSV download when… |
|---|---|
| You want a finished census loaded directly into the on-prem CTFSWeb MySQL. | You need the historical 10-column ctfsweb measurement CSV for a tool that still parses it. |
| You can run a `.sql` script against the destination MySQL. | You only have access to a CSV-consuming workflow. |
| You want destination identity reconciliation (Tree/Stem matching) handled for you. | You're going to re-derive identity downstream yourself. |
| The destination already has Suzanne's `creating_ViewFullTable.sql` helpers installed. | You don't need a ViewFullTable refresh. |

The Publish flow is the recommended path for any new CTFSWeb load.

## Prerequisites

The export will refuse to publish unless every one of these is true.

### 1. The census must be "Finished"

Every measurement included in the export must satisfy:

- `IsValidated = TRUE` (not `FALSE`, not `NULL`)
- No unresolved row in `measurement_error_log`
- `StemGUID IS NOT NULL` (no app-side failure markers)
- Joined `stems` and `trees` are active (`IsActive = 1`, `DeletedAt IS NULL`)
- Every `cmattributes.Code` resolves to an active `attributes` row
- Species row has `SpeciesCode`, `SpeciesName`, `Genus`, and `Family` set
- All destination-bound strings fit CTFS column widths (Tag ≤ 10, StemTag ≤ 32, QuadratName ≤ 8, etc.)

If anything fails, the modal returns a structured list of disqualifying `CoreMeasurementID`s grouped by failure kind. Fix them in the app and retry.

### 2. The destination must have CTFSWeb's reporting helpers installed

Suzanne's `creating_ViewFullTable.sql` installs `CreateFullView`, `CreateViewTaxonomy`, and supporting procedures in the `ctfsweb_webuser` schema. Source it **once per destination MySQL server**:

```bash
mysql -u root -p < creating_ViewFullTable.sql
```

The artifact probes for `ctfsweb_webuser.CreateFullView` before doing any work; if the procedure is missing it aborts with an install hint and zero data lands.

### 3. The destination must be on the post-`DBCHANGES2014f` schema

Specifically, `DBHAttributes` must no longer have a `CensusID` column. The artifact probes for this too and aborts with `apply DBCHANGES2014f.sql to the destination first` if it finds the legacy column.

### 4. Your account must have export permission

- **Lead technicians**: can publish censuses for schemas in their session-scoped site list. Normal exports only — reload options are hidden.
- **Admins** (`global` / `db admin`): can publish for any schema and can generate reload artifacts (see [Reload semantics](#reload-semantics)).

## Step by step

1. Sign in and select a site, plot, and validated census.
2. Navigate to **Measurements Hub → Census Overview**.
3. Click **Publish census** in the page header (top right).
4. The modal shows the source app context (schema, PlotID, CensusID, PlotCensusNumber) you're exporting from.
5. Enter the **Destination CTFS Plot ID** that this census should attach to in CTFSWeb. Most sites use `1`. The destination plot must already exist; the artifact will refuse to publish if it doesn't.
6. (Admins only) Optionally tick **Allow reload** and/or **Dry run** — see [Reload semantics](#reload-semantics).
7. Click **Download SQL artifact**. The file downloads as `ctfs-export-<destPlot>-<census>-<timestamp>.sql`.
8. Run the file against the destination MySQL (see [Running the artifact](#running-the-artifact)).

## Options

| Option | Who can use it | What it does |
|---|---|---|
| Destination CTFS Plot ID | All | Names which CTFSWeb `Plot.PlotID` this census attaches to. Must already exist on the destination. |
| Allow reload | Admins | The artifact DELETEs existing `DBHAttributes` + `DBH` rows for the target census before re-inserting. Identity rows (`Tree`, `Stem`) are **never** deleted; orphan counts are reported instead. |
| Dry run | Admins | Generates a preview-only artifact that runs the reload-impact SELECTs and then `ROLLBACK`s. Useful to confirm what a reload would touch before doing it. |

:::caution
**Allow reload is destructive.** It removes every `DBH` and `DBHAttributes` row for the target census on the destination. Use Dry run first if you're not certain about scope.
:::

## What's in the artifact

The generated `.sql` file is a single self-contained script.

```
-- BEGIN HEADER (metadata: timestamp, source/destination IDs, row counts, options)
-- END HEADER

DROP PROCEDURE IF EXISTS csv_to_sql_v2_load_<destPlot>_<slug>_<hash>;
DELIMITER //
CREATE PROCEDURE csv_to_sql_v2_load_<destPlot>_<slug>_<hash>()
main: BEGIN
  -- DECLAREs and EXIT HANDLER for rollback
  -- GET_LOCK on (destPlot, census) — concurrent exports serialize here
  START TRANSACTION;

  -- Stage 0a: destination schema probes
  --   DBHAttributes.CensusID absent? → continue, else SIGNAL
  --   ctfsweb_webuser.CreateFullView installed? → continue, else SIGNAL
  -- Stage 0:  Census row exists for (destPlot, plotCensusNumber)? → continue, else SIGNAL
  -- Stage 0b: (reload only) DELETE DBHAttributes + DBH for the target census
  -- Stage 1:  temporary InnoDB staging tables, populated with all rows
  -- Stage 2:  destination identity lookup (Quadrat, Family→Genus→Species→SubSpecies, Tree, Stem, TSMAttributes)
  -- Stage 2c: Suzanne normalization (HOM=1.3 for new stems where DBH>0 AND HOM IS NULL)
  -- Stage 5:  destination contract checks — per-row error reasons, SIGNAL on any failure
  -- Stage 6:  bulk insert new Tree rows (plot-aware: respects Panama-style overlapping tags)
  -- Stage 7:  bulk insert new Stem rows
  -- Stage 8:  bulk insert DBH rows
  -- Stage 9:  bulk insert DBHAttributes rows
  -- Stage 10: final tally

  COMMIT;
  DO RELEASE_LOCK(...);
END //
DELIMITER ;

CALL csv_to_sql_v2_load_...();
DROP PROCEDURE csv_to_sql_v2_load_...;

-- After the load procedure (DDL — runs outside the transaction):
CALL ctfsweb_webuser.CreateFullView(DATABASE(), 'ViewFullTable');
SELECT 'ViewFullTable rebuild' AS scope, 'completed' AS status;
```

A few invariants worth knowing:

- **The procedure name has a deterministic 8-character hash suffix** derived from the source schema, source plot/census, and destination plot/census. Two operators exporting the same census from the same schema get the same suffix. Two operators exporting from *different* source schemas to the same destination get different suffixes — so their `DROP PROCEDURE` / `CREATE PROCEDURE` statements never collide.
- **The body is byte-deterministic for the same input.** Only the header timestamp varies between runs. This is useful for ticket attachments and golden-file regression testing.
- **A `GET_LOCK` on `ctfs-export:<destPlot>:<plotCensusNumber>`** ensures two operators exporting to the same destination census serialize. The second one fails fast with "Another ctfs-sql export is running."
- **`ViewFullTable` rebuild runs *outside* the load procedure.** `CreateFullView` does `DROP/CREATE TABLE` (DDL → implicit commit) which can't live in the load transaction. If the rebuild fails after a successful load, the data is committed and the operator can re-run the rebuild step independently.

## Running the artifact

```bash
# One-time setup per destination MySQL server (see Prerequisites):
mysql -u <user> -p < creating_ViewFullTable.sql

# Each time you publish:
mysql -u <user> -p <destination_database> < ctfs-export-<destPlot>-<census>-<timestamp>.sql
```

You'll see:

- One or more `SELECT` result sets while the script runs (Stage 0b reload counts, Stage 10 final tally, post-load rebuild sentinels).
- An `ERROR 1644 (45000): ...` line if anything fails. The procedure rolls back automatically; no partial data is committed.

The final tally looks like:

```
+-------------------+------------+------------+
| measurement_rows  | tree_count | stem_count |
+-------------------+------------+------------+
|             10256 |       4823 |       9974 |
+-------------------+------------+------------+
```

And the rebuild sentinel:

```
+----------------------+-----------+
| scope                | status    |
+----------------------+-----------+
| ViewFullTable rebuild| completed |
+----------------------+-----------+
```

If you see both, the publish succeeded.

## Reload semantics

Reload mode (`allowReload=true`) is the way to re-publish a census after fixing data in the app. The artifact will:

1. Verify the destination Census row still exists.
2. Snapshot which stems and trees would become orphans if this census's DBH rows were deleted.
3. SELECT how many `DBHAttributes` and `DBH` rows the reload will delete.
4. `DELETE` those rows.
5. SELECT post-delete orphan counts (stems with no remaining DBH; trees with no remaining stems).
6. Continue with normal insert path.
7. Rebuild `ViewFullTable`.

**Identity rows are never deleted.** A reload only removes DBH and DBHAttributes for the target census; if a tree or stem is left with no surviving measurements, the operator decides whether to prune it manually outside this script. Identity-table deletion would risk breaking FK references that exist outside the script's knowledge.

### Dry run (`reloadDryRun=true`)

Dry-run mode wraps the reload preview in a `SAVEPOINT` and `ROLLBACK TO SAVEPOINT`. The script:

- Runs the reload-impact `SELECT`s for real (counts you'd actually see in a reload).
- Performs the `DELETE`s so the orphan counts are accurate.
- Rolls everything back so no destination row changes.
- Skips Stages 1–10 entirely (no inserts attempted).
- Skips the `ViewFullTable` rebuild.

The net effect on the destination is zero. The net effect on the operator's screen is four result sets describing the impact of a real reload.

## Troubleshooting

### "Census is not finished"

The modal returns a structured list of failure kinds with the first 20 disqualifying `CoreMeasurementID`s per kind. Each kind maps to one of the preconditions:

| Kind | What to fix |
|---|---|
| `not-validated` | Re-run validation in the app for those rows. |
| `unresolved-error` | Resolve the rows in **Measurements Hub → Errors**. |
| `no-stem-guid` | App-side failure marker. The row needs re-ingestion. |
| `inactive-join` | The joined stem or tree is soft-deleted. Restore or remove the measurement. |
| `unknown-attribute-code` | One or more `cmattributes.Code` values don't exist in `attributes`. Add the codes or fix the measurement. |
| `missing-taxonomy-fields` | The species row is missing `SpeciesCode`, `SpeciesName`, `Genus`, or `Family`. Fix in **Fixed Data Input → All Taxonomies**. |
| `string-too-long` | A destination-bound value exceeds the legacy CTFS column width. Shorten the offending field. |
| `zero-exportable-rows` | No measurements remain after applying all filters. Verify the census, plot, and validated state. |

### "DBHAttributes still has a CensusID column"

The destination is on the pre-`DBCHANGES2014f` schema. Source `DBCHANGES2014f.sql` against the destination once, then retry.

### "ctfsweb_webuser.CreateFullView missing"

Suzanne's helpers aren't installed on the destination. Source `creating_ViewFullTable.sql` once per destination server.

### "Expected exactly one Census row for PlotID=N, PlotCensusNumber=X; found 0"

The destination doesn't have a Census row for the `(destinationPlotID, plotCensusNumber)` pair. Create the destination Census row in CTFSWeb first, then retry. The artifact never creates Census rows for you.

### "Another ctfs-sql export is running for this destination (plot, census)"

A concurrent export holds the lock. Wait for it to finish or release the lock manually with `SELECT RELEASE_LOCK('ctfs-export:<destPlot>:<plotCensusNumber>');` from a session that holds it.

### "Validation failed; see prior SELECT for per-row details"

A Stage 5 contract check failed. The SELECT result sets immediately preceding the error list every offending row with a `CoreMeasurementID`, the natural keys, and an `Errors` column describing each failure. Common causes:

- **Ambiguous tree key**: The destination has multiple `Tree` rows with the same `(Tag, SpeciesID, SubSpeciesID)` in the target plot. Dedup or fix the destination.
- **Ambiguous stem key**: The destination has multiple `Stem` rows with the same `(TreeID, StemTag)`. Dedup or fix.
- **Taxonomy not uniquely resolved**: The taxonomy chain on the destination resolves to multiple `(SpeciesID, SubSpeciesID)` pairs. Usually a stale `CurrentTaxonFlag`. The error message includes the conflicting destination SpeciesIDs.
- **Unknown quadrat / Unknown TSMCode**: The destination is missing a referenced quadrat or TSM attribute. Add it first.
- **Empty StemTag collides with another stem under same Tree+Quadrat**: Two stems with `StemTag = ''` under the same tree and quadrat would collide on the destination. Assign distinct tags.

## Audit

Every Publish census action is logged with:

- Authenticated user id
- Source schema, source app PlotID, source app CensusID, source PlotCensusNumber
- Destination CTFS PlotID
- Measurement and attribute row counts
- Options (`allowReload`, `reloadDryRun`)
- Generated procedure name, lock name, download filename

These appear in the app's `ailogger.info` stream and can be correlated with the artifact's header comment (which includes the same metadata) when investigating an incident.
