Standing up a new ForestGEO site used to be a sequence of manual SQL steps run against the database by an engineer. As of **May 2026**, provisioning happens entirely in the app through a guided wizard at **`/admin/provision`** — admins fill out four short forms and the system handles the schema creation, table seeding, validations, catalog registration, and quadrat generation in one transactional run.

:::note
Site provisioning is restricted to users with the **global** role. If you don't see the option in the admin sidebar, you don't have access.
:::

### 1 · Site

The first step captures the catalog-level metadata.

| Field | What it is | Notes |
|---|---|---|
| **Site name** | Display name shown in the site selector | e.g. `Rabi` |
| **Schema name** | The MySQL schema that will hold this site's data | Lowercase letters, numbers, underscores only. Must start with `forestgeo_` — e.g. `forestgeo_rabi` |
| **Location** | Locality | Free text |
| **Country** | Country name | Free text |
| **Subquadrat dimensions** | `SQDimX` × `SQDimY` | Defaults to 5 × 5 |
| **Default DBH unit** | `mm` (most common) | Used for upload parsing |
| **Default HOM unit** | `m` (most common) | Used for upload parsing |
| **Double data entry** | Whether measurements must be entered twice to be accepted | Off by default |

:::caution
**Schema name is permanent.** Once provisioning succeeds, you can't rename the schema from the UI. Pick a slug that will read well years from now.
:::

### 2 · Plot

Define the first plot for the new site. You can always add more plots later — this is just the one that gets created during provisioning.

| Field | Notes |
|---|---|
| **Plot name** | Must be unique across the site |
| **Dimensions X × Y** | Plot size in metres (or whatever you set as the default coordinate unit) |
| **Area** | Computed automatically from dimensions |
| **Global X / Y / Z** | The plot's origin coordinates — used by location-based validations |
| **Plot shape** | `square` by default; the only validated geometry today |
| **Default units** | Dimension, coordinate, area, DBH, and HOM units. Defaults to metric |

### 3 · Quadrats

Two modes are available:

<CardGrid>
  <Card title="Grid mode (regular plots)">
    Specify a quadrat size (e.g. 20 × 20 m) and the wizard will tile the plot automatically. The plot dimensions must be evenly divisible by the quadrat size — if they aren't, switch to CSV mode.
  </Card>
  <Card title="CSV mode (irregular plots)">
    Upload a CSV of quadrats with `startX`, `startY`, `dimensionX`, `dimensionY`, and `quadratName` columns. The wizard validates that no quadrats overlap and that all of them fall inside the plot.
  </Card>
</CardGrid>

Both modes show a live preview of the layout and surface any geometric problems (overlaps, out-of-bounds rows, non-divisible plot dimensions) before you can advance.

:::tip
For most established ForestGEO sites the standard 20 × 20 m quadrat grid is correct — start with Grid mode. CSV mode exists for sites that already have an irregular historical layout you want to preserve.
:::

### 4 · Review

The final step summarises everything you entered. Submit to start the run.

## Watching and managing runs

Visit **`/admin/provision/runs`** for a table of every provisioning run with its status and timestamps.

| Status | Meaning |
|---|---|
| **running** | Currently in progress — open the run page to watch step-by-step progress |
| **completed** | Finished successfully; site is live |
| **failed** | A step errored. Open the run to see which step and the error message |
| **aborted** | A user clicked Abort during the run |

A heartbeat worker keeps long-running provisioning alive even across app server restarts; if a run is mid-flight when the server bounces, it will be picked back up automatically.

### Retry and abort

If a run **fails**, the failure detail tells you exactly which step errored. After fixing the underlying cause (e.g. a database connectivity blip), you can retry from the failed step rather than starting over — the steps that already succeeded are not re-run.

**Abort** is available while a run is `running`. It tears down anything that was created in that run transactionally — there are no orphaned half-provisioned schemas left behind.

## What still needs to happen manually after provisioning

The wizard sets up the **structure**; the **content** still needs to be loaded:

1. **Taxonomy** — load family, genus, species, and any species limits via the standard upload flow.
2. **Personnel & roles** — add the researchers who will be working in this site.
3. **Attributes** — load the site's attribute/stem code definitions.
4. **User access** — for non-global users, add `catalog.usersiterelations` rows so they can see the new site (this is on the admin Users → Sites page).
5. **Measurements** — once supporting data is in place, you can upload the first census's measurements through the normal upload flow.

## Legacy CTFS import

If the new site is being created from a **CTFS legacy `stable_*` schema**, do **not** use the provisioning wizard. The CTFS path is destructive to the target schema and runs a different migration sequence. Coordinate with engineering — there's a separate CLI runner that handles legacy imports.
